Project Overview

Think of this app like a digital stadium. Fans walk in the front door to watch matches and check scores. Captains have a special locker room where they manage their teams. And admins sit in the control room, running everything behind the scenes.

The TCL × Arsenal Road to Greatness 2026 app is a complete tournament management platform for a grassroots football (soccer) tournament in the UAE. It's sponsored by TCL Electronics and officially connected with Arsenal Football Club.

🎯 What Does It Do?

  • For Fans: Browse matches, check live scores, view standings, explore teams, watch highlights, and learn about prizes — all from their phone.
  • For Captains: Manage their team rosters, submit kit sizes, and sign agreements.
  • For Admins: Control everything — create matches, manage teams, publish content, send notifications, and run the entire tournament.

System Architecture

Imagine a restaurant. The customer (user) sits at a table (browser). The waiter (Next.js) takes orders and brings food. The kitchen (API routes) prepares the food. The pantry (database) stores all the ingredients. And the photo studio (Cloudinary) handles all the pretty pictures of food for the menu.

Our app uses a monolithic architecture — that means everything lives in one place (one Next.js app). This keeps things simple! The frontend (what you see) and the backend (what does the work) are both inside the same project.

Authentication Flow

Think of it like a concert wristband. You show your phone number at the gate (login). They text you a secret code (OTP). You tell them the code, and they give you a wristband (JWT token). That wristband lets you in, and different colored wristbands give access to different areas (roles).

We use phone-based OTP authentication. No passwords to remember! You just need your phone number. Here's exactly what happens step by step:

🔑 The Complete Auth Flow

flowchart TD
    A["📱 User opens /login"] --> B["📝 Enters phone number"]
    B --> C["📤 POST /api/auth/otp/start"]
    C --> D{"Phone valid?"}
    D -->|"No"| E["❌ Show error message"]
    D -->|"Yes"| F["🔢 Generate 6-digit OTP"]
    F --> G["💾 Store OTP in database\n(expires in 5 minutes)"]
    G --> H["📲 Send OTP to phone\n(SMS or console in dev)"]
    H --> I["📝 User enters 6-digit code"]
    I --> J["📤 POST /api/auth/otp/verify"]
    J --> K{"Code correct &\nnot expired?"}
    K -->|"No"| L["❌ Wrong code, try again\n(max 5 attempts)"]
    K -->|"Yes"| M["👤 Find or create User"]
    M --> N["🎫 Sign JWT token\n(using jose library)"]
    N --> O["🍪 Set HTTP-only cookie\n(token stored safely)"]
    O --> P{"What is their role?"}
    P -->|"FAN"| Q["🏠 Redirect to /"]
    P -->|"CAPTAIN"| R["👥 Redirect to /my-team"]
    P -->|"ADMIN"| S["🔧 Redirect to /admin"]

🛡️ How JWT Protects Pages

Once you're logged in, every time you visit a protected page, the middleware (a security guard at every door) checks your JWT cookie:

flowchart LR
    A["🌐 Request to any page"] --> B["🛡️ Middleware runs"]
    B --> C{"Has JWT cookie?"}
    C -->|"No"| D{"Is page protected?"}
    D -->|"No"| E["✅ Allow access\n(public page)"]
    D -->|"Yes"| F["🔒 Redirect to /login"]
    C -->|"Yes"| G["🔍 Verify JWT with jose"]
    G --> H{"Token valid?"}
    H -->|"No"| F
    H -->|"Yes"| I{"Role authorized\nfor this page?"}
    I -->|"No"| J["🚫 403 Forbidden"]
    I -->|"Yes"| K["✅ Allow access"]

Database Schema

The database is like a giant filing cabinet. Each drawer (table/model) holds a specific type of document. The "Teams" drawer has cards for each team. The "Matches" drawer has sheets for each game. Some documents have sticky notes pointing to other drawers — those are relationships (like "this match involves Team A and Team B").

We have 27+ database models organized into logical groups. The database is PostgreSQL hosted on Neon (a serverless database), and we talk to it using Prisma ORM (which translates our JavaScript code into SQL).

Running a match day is like being a movie director on set. Before shooting, you check all the equipment (pre-match). During filming, you call the shots (live scoring). After filming, you review the footage and share it with the world (post-match highlights).

    subgraph Morning["🌅 Morning Setup"]

        A4["📢 Send 'Match Day!' notification to fans"]

Glossary

Here are simple definitions for all the technical terms used in this documentation. We explain them like you're five — no jargon, just plain English!

API (Application Programming Interface)
🍽️ Like a waiter at a restaurant. You tell the waiter what you want (request), the waiter goes to the kitchen (server), and brings back your food (response). APIs let different parts of the app talk to each other.
JWT (JSON Web Token)
🎫 Like a wristband at a concert. Once you prove who you are at the gate, you get a wristband. Security guards just check your wristband without asking you to prove yourself again.
OTP (One-Time Password)
🔑 Like a temporary secret knock. It works once and expires quickly. You get a special code on your phone that you use to prove it's really you.
CRUD
📝 The four basic things you can do with data: Create (add new), Read (look at), Update (change), Delete (remove). Like adding a new contact, reading it, editing the number, or deleting it.
CMS (Content Management System)
📰 Like a newspaper editor's desk. It's where admins write and manage all the content that appears on the website — without needing to touch any code.
CDN (Content Delivery Network)
📬 Like having post offices in every neighborhood instead of one central one. Images and files are copied to servers around the world so they load fast no matter where you are.
ORM (Object-Relational Mapping)
🗣️ Like a translator. Our code speaks JavaScript, and the database speaks SQL. The ORM (Prisma) translates between them so they can understand each other.
Serverless
☁️ Like an on-demand taxi vs owning a car. Instead of running a server 24/7, the cloud spins up a mini-server just when someone needs it, then shuts it down. Pay only for what you use!
Middleware
🚧 Like a security checkpoint. Every request passes through it before reaching the page. It checks things like "are you logged in?" and "are you allowed here?"
Migration
🏗️ Like a renovation plan for your database. When you need to add a new room (table) or change the layout, you create a migration that safely makes those changes.
Seed
🌱 Like planting sample flowers in a new garden. Seeding fills an empty database with sample/test data so you have something to work with during development.
SSR (Server-Side Rendering)
🎁 Like gift-wrapping before delivery. The server prepares the complete HTML page before sending it to your browser, so it loads faster than building it piece by piece in the browser.
Cookie
🍪 A tiny note that the website saves in your browser. It remembers things like "this person is logged in" so you don't have to log in every time you visit a new page.
Enum
📋 A fixed list of options. Like a dropdown menu with only specific choices. For example, MatchStatus can ONLY be SCHEDULED, LIVE_FIRST_HALF, etc. — nothing else.
Schema
📐 The blueprint of your database. It defines what tables exist, what columns they have, and how they're connected — like an architect's plan before building a house.
Endpoint
🚪 A specific door in the API. Each endpoint is a URL that does one thing. /api/matches is the door for getting match data. /api/teams is the door for team data.
Cache
📸 Like taking a photo of a menu so you don't have to read it again. The app remembers data it already fetched, so it doesn't need to ask the server again right away.
Deployment
🚀 The process of putting your code on the internet so real people can use it. Like printing your book and putting it on store shelves for customers to buy.
PWA (Progressive Web App)
📱 A website that behaves like a phone app. You can add it to your home screen, and it works even with a shaky internet connection. No app store needed!
TypeScript
🏷️ JavaScript with labels on everything. It adds "type safety" — so if a function expects a number, it won't accidentally get a word. Catches mistakes before they become bugs.
Webhook
📞 Like a phone call from a friend. Instead of you checking if something happened, the system calls YOU automatically when something important happens.
Component
🧱 A reusable building block. Like LEGO pieces — you build a "MatchCard" component once, and then use it 31 times (once for each match) without rewriting the code.
Hook (React)
🎣 A special function that lets components "hook into" React features. Like useState (remember a value) or useEffect (do something when the page loads).
Route
🛣️ A path/address in the app. Like street addresses — /matches takes you to the matches page, /admin takes you to the admin area.
